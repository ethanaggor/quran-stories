# Spec

## Status

Ready for implementation.

This spec finalizes two UX refinements for the all-chapter interaction layer:

1. Replace the current `?` learning button with a Story Spark affordance that invites discovery.
2. Add a larger-screen reader frame so the same 9:16 mobile pages feel intentional on iPad and
   desktop without stretching the art.

This spec supersedes the current `?` affordance described in `plan.md` and implemented in
`public/index.html`.

## Requirements

- The reader remains a storybook first. Interactivity must feel like discovering a detail on a page,
  not opening help, taking a quiz, or entering a separate lesson mode.
- The interaction entry point must not use `?` as visible UI.
- The entry point must be icon-led, touch-friendly, and obvious enough for a young reader.
- The interaction entry point must preserve the existing three interaction kinds:
  `hotspots`, `choice`, and `sequence`.
- The bottom sheet interaction engine remains the runtime surface for all interaction kinds.
- The app keeps the existing image assets and page ratio. Images are not regenerated for this work.
- On mobile-width screens, the app remains full-bleed edge-to-edge.
- On iPad, laptop, and desktop screens, the app is centered in a 9:16 frame. The frame scales by
  available viewport height and never crops the page.
- No new third-party dependency is required for this work.
- All motion must use `transform` and `opacity`, and must respect `prefers-reduced-motion`.
- Completion state remains localStorage-based and keyed by `chapter.id + ":" + interaction.id`.

## UX Decisions

### Story Spark affordance

Replace the current `?` button with a circular Story Spark medallion.

The incomplete state:

- sits in the lower-right safe area, using the existing `.learn-btn` placement unless QA shows
  overlap on a specific page;
- renders an icon-only spark, not a text glyph;
- uses a warm gold and parchment material that belongs to the book palette;
- gives one short attention cue when the page becomes active;
- breathes very subtly while incomplete, with a long interval and no distracting constant bounce;
- has `aria-label="Open discovery moment"`;
- opens the existing interaction bottom sheet.

The first-time invitation state:

- on the first incomplete interaction opened in a browser session, the medallion may briefly expand
  a small attached label reading `Discover`;
- the label retracts after about 1600ms;
- the label must not persist while reading, must not overlap narrative text, and must not be shown
  for completed interactions;
- store this as session-only state, not localStorage.

The active/open state:

- hides the page spark while the bottom sheet is open, matching current behavior;
- the opening motion should feel like the spark is handing attention to the sheet:
  transform scale down to 0.94, fade out, sheet slides in.

The completed state:

- replaces the spark icon with a small check mark icon;
- removes pulsing/breathing motion;
- keeps the medallion visible but quieter, so the reader can reopen the moment without it asking for
  attention again;
- keeps the same localStorage completion behavior.

### Icon implementation

Do not add an icon dependency. Use small inline SVG strings generated by JS.

Add these helpers near the existing interaction helpers in `public/index.html`:

```js
function sparkIcon() {
  return `<svg class="learn-icon" viewBox="0 0 24 24" aria-hidden="true">
    <path d="M12 3l1.75 5.25L19 10l-5.25 1.75L12 17l-1.75-5.25L5 10l5.25-1.75L12 3Z"></path>
    <path d="M19 15l.75 2.25L22 18l-2.25.75L19 21l-.75-2.25L16 18l2.25-.75L19 15Z"></path>
    <path d="M5 4l.6 1.8 1.8.6-1.8.6L5 8.8 4.4 7 2.6 6.4l1.8-.6L5 4Z"></path>
  </svg>`;
}

function checkIcon() {
  return `<svg class="learn-icon" viewBox="0 0 24 24" aria-hidden="true">
    <path d="M6.5 12.5l3.2 3.2 7.8-8.4"></path>
  </svg>`;
}
```

The SVGs must be styled with CSS:

- `fill: none`
- `stroke: currentColor`
- `stroke-width: 1.8`
- `stroke-linecap: round`
- `stroke-linejoin: round`

### Story Spark markup

Update the `pageHTML` learning button generation from text content to icon content.

Target shape:

```js
const complete = p.interaction && isInteractionComplete(ch.id, p.interaction);
const learning = p.interaction
  ? `<button class="learn-btn ${complete ? 'done' : ''}" data-page="${i}" onclick="openInteraction(${i}, event)" aria-label="Open discovery moment">
      <span class="learn-symbol">${complete ? checkIcon() : sparkIcon()}</span>
      ${shouldShowDiscoverCue(ch.id, p.interaction) && !complete ? '<span class="learn-cue">Discover</span>' : ''}
    </button>`
  : '';
```

`shouldShowDiscoverCue` must be session-scoped. A simple module-level boolean is enough:

```js
let discoverCueShown = false;

function shouldShowDiscoverCue() {
  if (discoverCueShown) return false;
  discoverCueShown = true;
  return true;
}
```

If QA shows the label can appear on a bad page, remove the text cue and keep the icon animation only.

## Responsive Reader Frame

### Structural change

Wrap the current home and reader roots in an app shell and app frame.

Target body structure:

```html
<body>
  <div id="appShell">
    <main id="appFrame" aria-label="Stories from the Qur'an">
      <div id="home"></div>
      <div id="reader">
        ...
      </div>
    </main>
  </div>
</body>
```

After this change:

- `#home` and `#reader` are positioned relative to `#appFrame`, not the browser viewport.
- The existing full-bleed reader behavior remains inside the frame.
- The progress bar, tap zones, bottom sheet, hotspot markers, and Story Spark all stay inside
  `#appFrame`.

### CSS frame contract

Base mobile behavior:

```css
html, body {
  width: 100%;
  min-height: 100%;
  background: #0e0b1a;
  overscroll-behavior: none;
}

body {
  min-height: 100dvh;
  overflow: hidden;
}

#appShell {
  position: fixed;
  inset: 0;
  display: grid;
  place-items: center;
  background: #0e0b1a;
  overflow: hidden;
}

#appFrame {
  position: relative;
  width: 100vw;
  height: 100dvh;
  overflow: hidden;
  background: #0e0b1a;
}

#home,
#reader {
  position: absolute;
  inset: 0;
}
```

Larger-screen behavior:

```css
@media (min-width: 700px) and (min-height: 620px) {
  #appShell {
    padding: 16px;
    background: linear-gradient(180deg, #211a15 0%, #0f0d0c 100%);
  }

  #appFrame {
    width: min(calc(100vw - 40px), calc((100dvh - 32px) * 9 / 16), 560px);
    height: auto;
    aspect-ratio: 9 / 16;
    border-radius: 24px;
    box-shadow:
      0 28px 90px rgba(0, 0, 0, .48),
      0 0 0 1px rgba(255, 255, 255, .08),
      inset 0 1px 0 rgba(255, 255, 255, .08);
  }
}
```

Frame acceptance rules:

- At `390x844`, the frame is indistinguishable from the current full-screen mobile app.
- At `768x1024`, the app is centered, 9:16, and fully visible.
- At `1024x768`, the app is centered, 9:16, and fully visible.
- At `1440x900`, the app is centered, 9:16, and no wider than `560px`.
- No page image is stretched outside its 9:16 frame.
- No horizontal page scroll is introduced.
- The browser background is a quiet matte backdrop, not a decorative hero or landing page.

## Testing and Validation Strategy

- Do not add tests that enforce prose wording in Markdown docs.
- Do not add brittle screenshot assertions.
- Use `bun run validate:interactions` to protect structured interaction data.
- Use runtime browser QA to protect the actual interaction contract:
  - page framing;
  - touch target visibility;
  - interaction opening;
  - hotspot marker alignment;
  - choice feedback;
  - sequence completion;
  - desktop keyboard Escape behavior.
- Use visual screenshots as evidence for representative states:
  - mobile Story Spark;
  - iPad portrait frame;
  - iPad landscape frame;
  - desktop frame;
  - completed Story Spark state.

## Risks and Edge Cases

- The `Discover` cue may overlap page text on some pages.
  - If this happens, remove the text cue and keep the icon-led spark animation.
- Framing may affect fixed-position assumptions.
  - Convert reader roots to frame-relative positioning in one pass and QA all overlays.
- Safe-area handling may look too padded on framed iPad/Desktop.
  - Keep `env(safe-area-inset-*)` for mobile and verify desktop frame padding visually.
- The warm spark can become visually loud.
  - Keep motion short, remove infinite bounce, and use completed quiet state.
- A large desktop frame can make text feel too large.
  - Cap frame width at `560px`; do not scale fonts by viewport width.
- The bottom sheet may cover too much of the framed reader in landscape.
  - Keep the existing `max-height` behavior and verify sequence completion remains visible.

## Completion Criteria

- No visible `?` remains as the interaction affordance.
- Story Spark opens every existing interaction type.
- Completed interactions show a quiet check state.
- Mobile remains full-bleed.
- iPad and desktop use a centered, fully visible 9:16 frame.
- All validation gates pass.
- The implementation is committed and deployed to GitHub Pages.
